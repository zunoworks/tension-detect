# あなたのCLAUDE.md、矛盾してますよ — AIの「判断力」を作るたった1つの方法

CLAUDE.mdやCursor Rulesの矛盾を自動検出し、「いつどちらを使うか」の判断基準に変換するMCPサーバーを作った。`pip install tension-detect` で使える。追加APIコストはゼロ。

---

Claude Codeを使っていて、こんな経験はないだろうか。

- 「コードレビューは丁寧にやれ」と書いてある。「PRは早くマージしろ」とも書いてある
- AIが毎回違う判断をする。同じルールなのに
- ルールを追加するたびに、前のルールと噛み合わなくなる

**原因はシンプルだ。ルールが矛盾している。**

「丁寧にレビューしろ」と「早くマージしろ」は両方正しい。しかしAIはどちらに従えばいいか知らない。だから毎回ランダムに選ぶ。

ここで考えてほしい。料理人の世界には「塩を効かせろ」と「塩を控えろ」がある。どちらも正しい。プロの料理人は矛盾を解消しない。**「素材の味が弱いときは効かせる。強いときは控える」**と知っている。

これが判断力だ。

## AIにはルールがある。判断力がない。

CLAUDE.mdに10個のルールを書く。3個目と7個目が矛盾している。でもお前は気づかない。AIも気づかない。だから毎回ブレる。

tension-detectはこの矛盾を見つけて、判断基準に変える。

```
ルールA: コードレビューは丁寧にやれ
ルールB: PRは早くマージしろ
　↓
境界: 本番障害のhotfix → 即マージ。通常時 → 丁寧にレビュー。
シグナル: 障害チケットが開いているか？
```

2つの矛盾するルールが、1つの判断基準に変わった。

## 実際に試す

```bash
pip install tension-detect
claude mcp add tension-detect -- python -m tension_detect
```

Claude Codeで聞く:

> 「CLAUDE.mdの矛盾を見つけて判断基準にして」

AIが自動で:
1. ルールを解析
2. 矛盾を検出（サーバー側、LLM不要）
3. 境界条件を考えて提案（AIが自分の頭で考える。追加コストゼロ）
4. CLAUDE.mdに書き戻す

## 何が出てくるか

実際のCursor Rulesで検証した結果:

```
⚡「Ship features fast」vs「Review PRs thoroughly」
  → 境界: wait ↔ act

⚡「Keep PRs small」vs「Consider full architecture」
  → 境界: part ↔ whole

⚡「Don't break existing」vs「Refactor when complex」
  → 境界: preserve ↔ change
```

11ルールから5つの矛盾を検出。偽陽性ゼロ。全て意味のある対立。

## なぜ誰もやっていなかったのか

矛盾を**検出**する研究はある。要件定義の矛盾検出（精度99%）、RAGの矛盾チェック。

矛盾を**議論**させる研究もある。ヘーゲル弁証法LLM、認知的不協和AI。

矛盾を**解決**するシステムもある。AIエージェント間紛争解決。

しかし全て「矛盾を見つける」か「矛盾を消す」かのどちらか。

**矛盾を「使えるようにした」ツールはなかった。**

見つけた。でも消さなかった。代わりに「いつどちらを使うか」の判断基準を抽出した。これがtension-detect。

## 技術的に何が起きているか

MCPサーバーがルールを受け取り、2つの戦略で矛盾を検出する:

1. **キーワード重複**: 同じ話題について言っているルールを見つける
2. **方向対立**: 4つの対立軸（confirm↔execute, preserve↔change, wait↔act, part↔whole）で逆方向を指すルールを見つける

否定も正しく処理する。「Don't ship without tests」は「ship」ではなく「wait」と判定される。

**サーバーにLLMは載っていない。** 検出は全てヒューリスティック。境界条件の生成はクライアントLLM（お前が既に動かしているClaude Code）が自分の頭で考える。追加APIコストはゼロ。

英語・日本語バイリンガル対応。

## まとめ

- AIのルールは必ず矛盾する。それは悪いことではない
- 矛盾を消すのではなく、判断基準に変えろ
- tension-detectがそれを自動でやる
- 追加コストゼロ、2コマンドで導入
